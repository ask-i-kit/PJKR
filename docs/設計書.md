## 1. 全体アーキテクチャ

### 1.1 構成概要

```
┌──────────────────────────────────────────────┐
│ pjkr.exe (Go)                                          │
│                                                        │
│  ┌────────────┐   ┌──────────────────┐  │
│  │ HTTPサーバ        │   │ ストア(store)             │  │
│  │ 127.0.0.1:8760    │←→│  ・AppData(メモリ上)      │  │
│  │  ・静的配信(embed) │   │  ・排他制御(Mutex)        │  │
│  │  ・REST API       │   │  ・自動保存(debounce)     │  │
│  │  ・パスを開くAPI    │   │  ・世代バックアップ        │  │
│  └────────────┘   └──────────────────┘  │
│         ↑ [Phase 2] トレイ常駐・トースト通知・朝のスケジューラ │
└─────────┼────────────────────────────────┘
          │ http://127.0.0.1:8760
┌─────────┴───────────┐      ┌─────────────┐
│ ブラウザ(Edge)             │      │ data/pjkr.json      │
│  SPA(素のJS、フレームワーク無)│      │ data/backup/*.json  │
└─────────────────────┘      └─────────────┘
```

- **単一プロセス・単一利用者・単一データファイル**。同時編集を考えない前提で、サーバはデータ全体をメモリに保持し、変更のたびに非同期でファイルへ書き出す。
- フロントエンドは起動時に全データを1回取得し、以後は操作ごとに差分APIを呼ぶ(楽観的UI更新)。

### 1.2 リポジトリ構成

```
PJKR/
├─ go.mod / go.sum
├─ main.go                  … 起動・配線(ポート解決、ブラウザ起動、シグナル処理)
├─ internal/
│   ├─ store/
│   │   ├─ model.go         … 全エンティティ定義(JSONスキーマと1対1)
│   │   ├─ store.go         … ロード/保存/排他/ID採番
│   │   ├─ backup.go        … 世代バックアップ
│   │   └─ migrate.go       … データバージョン移行
│   ├─ api/
│   │   ├─ server.go        … ルーティング、静的配信、共通ミドルウェア
│   │   ├─ handlers.go      … エンティティCRUD
│   │   └─ open.go          … ローカルパスを開くAPI
│   └─ busday/              … [Phase 2] 営業日計算(祝日データ同梱)
├─ web/                     … go:embed でexeに同梱
│   ├─ index.html
│   ├─ app.js               … エントリ(ルータ・ショートカット・undo)
│   ├─ api.js / state.js    … API呼び出し・クライアント状態
│   ├─ views/               … 画面ごとのモジュール(dashboard.js, tasks.js, …)
│   └─ style.css
├─ docs/
│   ├─ 要件定義書.md
│   └─ 設計書.md
└─ data/                    … 実行時に生成(gitignore)
```

- ES Modules をそのまま使い、ビルドツール(npm等)は導入しない。`web/` を直接編集して `go build` だけで完結させる。

### 1.3 起動シーケンス

1. `data/pjkr.json` をロード(無ければ初期データを生成)。データバージョンを確認し、古ければ移行(§4.4)
2. 起動時バックアップを作成(§4.3)
3. ポート解決: 既定8760。使用中なら `GET /api/ping` を呼び、**PJKR自身が応答したら二重起動**と判断して既定ブラウザでURLを開いて終了。PJKR以外が使用中なら8761〜8779を順に試す
4. HTTPサーバ起動(127.0.0.1のみバインド。要件N7)
5. 既定ブラウザで `http://127.0.0.1:{port}/` を開く
6. [Phase 1] コンソールウィンドウを表示したまま常駐(閉じる=終了)。[Phase 2] `-H windowsgui` +トレイ常駐に置き換える

### 1.4 技術選定(確定)

| 項目 | 選定 | 備考 |
|---|---|---|
| Go | 1.22以上 / 標準ライブラリ中心 | `net/http` の `ServeMux`(メソッド付きパターン)を使用 |
| 外部依存(Phase 1) | なし(標準ライブラリのみ) | exe実行可否の検証(未確定#7)を最小構成で行うため |
| 外部依存(Phase 2以降) | excelize(Excel)、systrayライブラリ、トースト通知 | Phase 2着手時に選定・追加 |
| フロントエンド | 素のJS(ES Modules)+ハッシュルーティング | フレームワーク・CDN不使用(要件N3) |
| ビルド | `go build -o pjkr.exe` | 配布はexe 1つ+data/フォルダ |

## 2. データ設計

### 2.1 JSONスキーマ(pjkr.json)

Phase 1で使わないフィールドも含め、**スキーマは要件v2.6の全体像で最初から定義**する(後の移行を減らすため)。日付は `"2026-07-12"`、日時は RFC3339 文字列。

```jsonc
{
  "version": 1,
  "counters": { "project": 3, "task": 120, "inbox": 15, "note": 8, "contact": 5, "issue": 2, "risk": 0, "milestone": 4, "report": 0 },
  "settings": {
    "waitAlertBizDays": 5,        // 返答待ち→催促候補(営業日)
    "askedAlertBizDays": 3,       // 被依頼→要回答(営業日)
    "notifyTime": "09:00",        // [P2] 朝の通知時刻
    "notifyEnabled": true,        // [P2]
    "customHolidays": []          // [P2] "YYYY-MM-DD" の配列
  },
  "projects": [{
    "id": "p1", "name": "基幹更新PJ", "status": "active",   // planned/active/onhold/done
    "start": "2026-04-01", "end": "2027-03-31",
    "summary": "", "vendor": "ベンダーA", "archived": false,
    "links": [{ "id": "l1", "label": "基本設計書", "kind": "file",   // file/folder/url/mail
                 "target": "\\\\share\\pj\\設計書.xlsx", "group": "設計書", "memo": "" }]
  }],
  "inbox": [{ "id": "i1", "text": "ベンダーAに見積依頼する", "createdAt": "..." }],
  "tasks": [{
    "id": "t1", "projectId": "p1", "parentId": null,
    "title": "見積依頼", "detail": "", "start": null, "due": "2026-07-20",
    "priority": "mid",             // high/mid/low
    "status": "todo",              // todo/doing/waiting/done/hold
    "progress": 0,
    "assignee": null,              // null=自分 / contactId=他部門・担当者
    "contactId": null,             // ボールの相手(返答待ち/被依頼で使用)
    "waitSince": null,             // 返答待ち開始日
    "replyHope": null,             // 回答希望日
    "nudges": [],                  // 催促日の配列 ["2026-07-18"]
    "isAsked": false,              // 被依頼フラグ(相手から受けた仕事)
    "askedOn": null,               // 被依頼の発生日
    "answer": "",                  // [P2] 回答概要(回答管理用)
    "checkMemo": "",               // [P2] 判断保留: 何を確認すれば回答できるか
    "todayOn": null,               // 「今日やる」ピック対象日 "2026-07-12"
    "links": [],                   // 関連リンク(projects.links と同形式)
    "logs": [{ "at": "...", "text": "電話で仕様を確認した" }],
    "createdAt": "...", "updatedAt": "...", "doneAt": null
  }],
  "notes": [{ "id": "n1", "projectId": "p1", "title": "7/10定例の決定事項",
              "body": "…", "pinned": false, "links": [],
              "createdAt": "...", "updatedAt": "..." }],
  "contacts": [{ "id": "c1", "name": "山田", "org": "ベンダーA", "memo": "" }],
  "milestones": [], "issues": [], "risks": [], "reports": []   // [P2以降]
}
```

**設計判断:**
- ID は `種別接頭辞+連番`(`t42` 等)。`counters` で採番し、課題番号(F6-5)にも流用できる
- タスクの「返答待ち」「被依頼」「今日」は独立フィールドの組合せで表現(要件v2.4の統合方針どおり)。状態遷移で `waitSince` 等を自動設定する(§5.2)
- 削除は物理削除。Undo(§6.3)が復元手段になる

### 2.2 サイズ見積り

タスク10,000件×平均500B ≒ 5MB。メモリ保持・全件ロード・全再描画とも問題ない規模(要件N8)。
JSONの読み書きは100ms未満想定。保存はdebounceで頻度を抑える。

## 3. API設計

すべて `Content-Type: application/json`。127.0.0.1限定のため認証なし。
CSRF対策として `Sec-Fetch-Site` が `same-origin`/`none` 以外のリクエストは拒否する(他サイトのページからlocalhostへの書き込みを防ぐ)。

| メソッド/パス | 内容 |
|---|---|
| `GET /api/ping` | 識別応答 `{"app":"pjkr","version":"..."}`(二重起動判定用) |
| `GET /api/state` | 全データ+設定を返す(起動時に1回) |
| `POST /api/{entity}` | 作成。entity = `projects/tasks/inbox/notes/contacts`。作成された実体を返す |
| `PATCH /api/{entity}/{id}` | 部分更新(送られたフィールドのみ)。**更新前の値(before)と更新後の実体を返す** → クライアントUndo用 |
| `DELETE /api/{entity}/{id}` | 削除。**削除された実体(子タスク含む)を返す** → Undo用 |
| `POST /api/tasks/{id}/log` | 作業ログ追記 `{text}` |
| `POST /api/inbox/{id}/convert` | Inbox変換 `{to:"task"|"note", fields:{...}}`。変換先を作成しInboxから削除(1トランザクション) |
| `POST /api/open` | `{target}` を検証しOSで開く(§5.4) |
| `POST /api/backup` | 手動バックアップ実行(設定画面用) |

- 変更系APIは処理後に保存をスケジュール(§4.2)
- エラーは `{"error":"メッセージ"}` + 適切なHTTPステータス
- [Phase 2以降] `/api/export/*`(Excel)、`/api/search`(件数が問題になる場合。まずはクライアント内検索)

## 4. ストア設計

### 4.1 排他と整合性

- `sync.Mutex` で AppData 全体を保護(単一利用者のため粒度はこれで十分)
- ハンドラは「ロック→変更→保存スケジュール→アンロック→応答」の順

### 4.2 自動保存(要件N4)

- 変更後 **500msのdebounce** でフラッシュ(連続操作をまとめる)。加えて **最大3秒** で必ず書く
- 書き込みは atomic: `pjkr.json.tmp` に書いて `fsync` 後 `os.Rename`(要件N5)
- 終了シグナル(Ctrl+C/ウィンドウクローズ)で未保存分をフラッシュしてから終了

### 4.3 世代バックアップ(要件N5)

- **起動時**: `data/backup/pjkr-20260712-090000.json` を作成
- **日次**: その日最初の保存時にも作成
- 保持数: 直近**20世代**(超過分は古い順に削除)
- 設定画面から手動バックアップ・バックアップフォルダを開く操作を提供

### 4.4 マイグレーション(要件N13)

- `version` フィールドを見て `migrate.go` の関数列を順に適用(`v1→v2→…`)
- 移行前に必ずバックアップを取る

## 5. 主要ロジック設計

### 5.1 「今日」ビューの抽出(F7-1〜3)

```
自動候補(Phase 1):
  期限切れ   = status ∉ {done} かつ due < today
  今日期限   = status ∉ {done} かつ due = today
  [P2で追加] 催促候補・要回答(§5.3)、判断保留の滞留
手動ピック:
  todayOn = today のタスク(自動候補と重複したら自動側に表示)
昨日の残り:
  todayOn < today かつ status ≠ done
  → 「昨日の残り」欄に表示し、[今日もやる]=todayOnをtodayに更新 / [外す]=todayOnをnullに
```

### 5.2 タスクの状態遷移(F4関連)

```
todo ⇄ doing ⇄ waiting(返答待ち) → done
  └──────────┴─ hold(保留)

waiting へ変更時: contactId必須、waitSince=今日(編集可)、replyHope任意
waiting から戻す時: 回答内容を作業ログに残すようUIが促す(F4-10。スキップ可)
done へ変更時: doneAt=now。被依頼(isAsked)なら回答概要の入力を促す([P2] F14-1)
```

Phase 1では `waiting` は「状態として設定できる」まで(相手・経過表示はPhase 2のボール画面で本格化)。

### 5.3 営業日計算 [Phase 2](要件N14)

- 土日+同梱祝日テーブル(内閣府データから生成した `holidays.go`、年1回更新)+ `customHolidays`
- `bizDaysSince(from, today)` で経過営業日を算出。催促候補判定は「max(waitSince, 最終催促日)から waitAlertBizDays 営業日超過」

### 5.4 ローカルパスを開くAPI(F12-2, F3-11)

- 入力検証: `http(s)://` はブラウザ側で開く(API不要)。それ以外は
  1. パスとして評価(`\\share\...` / `C:\...`)。存在チェック(`os.Stat`)
  2. 存在すれば `explorer.exe <path>`(フォルダ)/ 関連付け起動 `cmd /c start "" <path>`(ファイル)ではなく **`explorer.exe <path>` に統一**(ファイルも関連付けで開く。コマンドインジェクションの余地をなくす)
  3. 存在しなければ 404 を返しUIでトースト表示
- `mail` 種別: 会社環境の方式が未確定(要件 未確定#9)のため、Phase 1では「文字列をそのまま `explorer.exe` に渡す(`outlook:` URIも可)+失敗時メッセージ」とし、検証結果で調整

### 5.5 ショートカット(要件N11)

| キー | 動作 |
|---|---|
| `Ctrl+I` | ヘッダーの書き留め欄へフォーカス(全画面) |
| `Ctrl+F` | 検索欄へフォーカス([P1]はタスク一覧内フィルタ、[P2]で横断検索に接続) |
| `Ctrl+Z` / `Ctrl+Y` | Undo / Redo(§6.3) |
| `Enter` | 書き留め欄で送信 |
| `Esc` | ダイアログ/詳細パネルを閉じる |

入力欄フォーカス中は `Ctrl+Z` をブラウザ標準(テキストundo)に譲る。

## 6. フロントエンド設計

### 6.1 構造

- **状態**: `state.js` がサーバから取得した AppData を保持。ビューは state から純粋に描画(仮想DOMなし、画面単位の再描画で十分な規模)
- **ルーティング**: `#/dashboard` `#/home/p1` `#/tasks` `#/inbox` `#/settings` のハッシュルータ
- **API**: `api.js` が fetch をラップ。変更系は「楽観的にstateを更新→API→失敗したらロールバック+トースト」

### 6.2 画面設計(Phase 1)

**ダッシュボード(起動時表示)**
```
┌ 今日 (7/12 土) ────────────────── 残り 4件 ┐
│ ⚠ 期限切れ                                        │
│   □ 仕様書レビュー   基幹更新PJ   期限7/10 (2日超過) │
│ ● 今日期限                                        │
│   □ 稟議書ドラフト   会計刷新PJ   期限7/12          │
│ ○ 今日やる(手動ピック)              [＋タスクから選ぶ] │
│   □ 設計書たたき台   基幹更新PJ   期限7/25    [外す] │
│ ▣ 昨日の残り                                      │
│   □ 環境手配の確認  [今日もやる] [外す]              │
├ 期限間近(3日以内) ──────────────────────┤
├ Inbox(未整理 3件) → [整理する] ─────────────┤
└ 案件サマリ(案件別のタスク消化状況) ─────────────┘
```

**タスク一覧**: 表形式+階層インデント。列=状態/件名/案件/担当/期限/優先度/今日。行クリックで右側に詳細パネル(編集・作業ログ・関連リンク)。上部にフィルタ(状態・案件・担当)と `Ctrl+F` の絞り込み
**Inbox**: 1行ずつ「→タスク化」「→メモ化」「破棄」。タスク化時は案件・期限・相手をその場で指定するミニフォーム
**案件ホーム**: 案件タブ(またはプロジェクト▼と連動)。リンク集(分類見出しごと、クリックで開く)+案件メモ(ピン留め上部固定、新しい順)+サマリ
**設定**: バックアップ操作、(P2以降の設定はプレースホルダ表示)

### 6.3 Undo/Redo(要件N12)

- クライアント側に **逆操作スタック** を持つ(最大50)
  - `PATCH` → 応答の before を使い逆PATCHを積む
  - `POST`(作成)→ 逆操作は DELETE
  - `DELETE` → 応答の実体(子含む)を使い逆操作は再POST(同一IDで復元。サーバは指定ID付きPOSTを許可)
  - Inbox変換 → 逆操作は「変換先DELETE+Inbox再POST」
- `Ctrl+Z` で逆操作を実行し、Redoスタックへ移す。トースト「削除を取り消しました」を表示
- ページリロードでスタックは消える(許容。直近の誤操作対策が目的)

## 9. Phase 2 実装(v0.2.0で追加)

### 9.1 追加パッケージ・依存

| パッケージ | 内容 |
|---|---|
| `internal/busday` | 営業日計算。祝日2025〜2027同梱(年1回更新)+独自休日。単体テストあり |
| `internal/notify` | go-toast によるトースト通知+朝の定時通知スケジューラ(30秒間隔で時刻監視、日1回送信、0件時は送らない) |
| `internal/startup` | スタートアップ登録。PowerShell経由で `PJKR.lnk` を作成/削除 |
| `internal/icon` | トレイアイコン(生成済みICOを埋め込み) |
| 外部依存 | getlantern/systray(トレイ常駐)、go-toast/toast(通知) |

### 9.2 追加API

| メソッド/パス | 内容 |
|---|---|
| `POST /api/settings` | 設定の部分更新(警告日数・通知・独自休日) |
| `POST /api/startup` | `{enabled}` でスタートアップ登録ON/OFF |
| `POST /api/notify-test` | 現在の件数でトーストを即時送信(設定画面の動作確認用) |
| `GET /api/state` | meta に `holidays`(祝日配布)・`startupEnabled` を追加 |

### 9.3 フロントエンド追加

- **画面**: ボール(相手/自分/相手別タブ)・判断保留箱・回答管理(Q&A)・課題リスク・横断検索(`Ctrl+F`)
- **共通ダイアログ**(`dialogs.js`): 返答待ち切替(相手必須)/回答受領(ログ記録)/回答して完了(回答概要)/催促記録
- **状態遷移の割り込み**: タスクの状態selectで waiting への切替・waitingからの復帰・被依頼の完了時に各ダイアログを挟む
- **一括登録**: 複数行貼り付け→工程テンプレート(既定「作成,検証」)で親子生成。1回のCtrl+Zで全取り消し

### 9.4 判定ルール(実装値)

- 催促候補: `max(waitSince, 最終催促日)` から **5営業日**(設定可)経過、または回答希望日超過
- 要回答: 被依頼の `askedOn` から **3営業日**(設定可)経過、または期限超過
- 判断保留の滞留: `askedOn` から **5営業日**(設定可)経過
- ダッシュボード「今日」の自動候補にこれらを合流。残件数にも含める

## 10. Phase 3 実装(v0.3.0で追加)

### 10.1 報告(F8)

- **下書き生成はクライアント側**(`views/report.js`)。対象週(月曜起点)の
  完了タスク(`doneAt`)・週内の作業ログ・未クローズ課題・要回答・来週期限タスク・返答待ち・リスクから
  案件別に「今週の実績/今週の課題/来週の予定/リスク・ボール状況」を構成する
- 編集は案件×4区分のテキストエリア。保存は `reports[]` エンティティ(週開始日をキーに上書き)。過去週報はセレクタで再読込
- **報告資料モード**: 整形表示+印刷(`@media print` でヘッダー・サイドバーを非表示)
- **Copilot用コピー**: 清書指示文+Markdown風の構造化テキストをクリップボードへ(外部通信なし)
- **Excel出力**: `POST /api/export/report` に編集後の内容を送り、excelizeで「案件/区分/内容」形式のxlsxを生成
  (現行様式が確認できたら(未確定#1)この出力関数だけ差し替える)

### 10.2 データ入出力(F9)

| メソッド/パス | 内容 |
|---|---|
| `GET /api/export/{tasks\|issues\|balls\|qa}` | 一覧のxlsxダウンロード(ボールは経過営業日つき、Q&AはFAQ素材) |
| `GET /api/template/{tasks\|issues}` | インポート用テンプレート(ヘッダー+記入例) |
| `POST /api/import/{tasks\|issues}` | multipartでxlsxを受け取り一括作成。案件名・担当名はマスタと突合し、行単位のエラーを返す。作成IDを返しクライアントがUndo登録(Ctrl+Zで全取消) |
