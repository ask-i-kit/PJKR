## 1. 背景・目的

### 1.1 背景

システム部門の業務では、ベンダー管理・発注、社内調整・企画、保守運用など性質の異なる案件を
4〜10件並行して担当しており、個人での状況把握の負担が大きい。
ヒアリングの結果、既存の管理方法(Excel・メモ・記憶)には以下の問題があることを確認した。

| # | 問題 | 内容 |
|---|---|---|
| 1 | **返答待ちの追跡漏れ** | ベンダーや他部署に依頼・質問したまま、回答の催促を忘れる(ボール管理ができていない) |
| 2 | 全体像の把握に時間がかかる | 「今何がどうなっているか」を思い出すのにファイルやメールを探し回る |
| 3 | 報告資料の作成負担 | 週次の案件別週報のたびに情報をかき集めてExcelに転記している |
| 4 | 期限・約束の忘れ | 期限の直前・事後に気づくことがある |
| 5 | 記録が残らない発生源 | タスクがメール・Teams・口頭/電話から発生するため、書き留める前に流れてしまう |
| 6 | **ベンダーからの質問対応の負担** | 回答の根拠(過去の経緯・資料)を探すのに時間がかかる。割り込みで作業が中断される。回答を待たせたり忘れたりすることがある |

### 1.2 目的

担当案件の「タスク(返答待ち・被依頼を含む)」「スケジュール」「課題・リスク」を1つのツールに集約し、

1. **ボールの所在を双方向で追跡する**(最重要)。自分→相手の依頼は催促漏れを、相手→自分の質問・依頼は回答忘れをなくす
2. **日々の状況把握を数秒で**行えるようにする
3. **週報の作成負担を減らす**(記録から自動で下書きを生成し、Excelで出力する)
4. **発生した瞬間のタスクを2秒で書き留められる**ようにする(割り込み時はInboxに書き留めて自分の作業に戻れる)
5. **過去の経緯・回答を検索で引ける**ようにする(質問対応で調べ直す時間を減らす)

ことを目的とする。

### 1.3 ゴール(成功基準)

- 依頼から5営業日回答がない案件が、自動的に「催促候補」として目に入る
- 自分が回答を持ち帰った質問が、放置される前に「要回答」として目に入る
- 過去に答えた内容・経緯が、キーワード検索で数秒で見つかる
- 毎朝、その日の優先タスク・要注意項目が30秒以内に把握でき、自動候補+手動ピックで「今日やること」リストを数分で組める
- 1つの仕事は1回の登録で最後まで追える(タスクと依頼を別々に登録する二重管理が発生しない)
- 週次報告のExcelが、下書き自動生成+微修正のみで完成する
- ツールを開き忘れていても、Windows通知で重要な期限・催促候補に気づける
- 会社PCで追加ソフトのインストールなしに動作する

## 2. 利用者・利用シーン

| 項目 | 内容 |
|---|---|
| 利用者 | 本人1名(システム部門所属) |
| 業務の性質 | ベンダー管理・発注、社内調整・企画が中心(保守運用の定常業務は本ツールの対象外) |
| 並行案件数 | 4〜10件 |
| 同時利用 | なし(共有・同時編集は対象外) |
| 利用環境 | 会社PC(Windows / ブラウザはEdge想定)。ソフトのインストールは不可だが、Python・Goは利用可能。exeの実行は可能 |
| 利用頻度 | 毎日(朝の確認・随時更新)、週次(週報出力) |
| 利用スタイル | 常時開きっぱなしは想定しない。**開き忘れても通知で気づける**ことが必要 |

### 業務の基本フロー

日々の業務は概ね次の流れで進む。ツールはこのフローの各ステップを支える。

| ステップ | 内容 | 対応する機能 |
|---|---|---|
| 1. 受ける | Teams・Outlookで依頼・質問を受ける | Inbox(F2)、被依頼登録(F4)、判断保留箱(F13) |
| 2. 調べる | 判断材料のExcel・メール・設計書を探す | 案件ホーム(F12)、横断検索(F10)、回答管理(F14) |
| 3. 判断する | すぐ答えられないものは確認事項を整理して保留する | 判断保留箱(F13) |
| 4. 依頼する | ベンダーや社内へ依頼・質問する | 返答待ちへの切替(F4) |
| 5. 待つ | 回答を待ち、必要なら催促する | 催促候補(F4)、「今日」ビュー(F7) |
| 6. 報告する | 上司へ実績・課題・予定を報告する | 報告資料モード・週報(F8) |

### 主な利用シーン

1. **朝の段取り**: 通知またはツールを開き、「今日」ビューで自動候補(期限切れ・今日期限・催促候補・要回答)を確認し、加えて今日進めたいタスクを手動でピックして今日やることリストを組む
2. **発生時の書き留め**: メール・Teams・電話でタスクや依頼が発生したら、Inboxに1行メモで即座に投げ込む
3. **依頼したとき**: ベンダー・他部署に何かを依頼したら、そのタスクの状態を「返答待ち(相手ボール)」に切り替える。以後ボールの所在が自動で追跡される
4. **質問を受けたとき**: ベンダーからの質問はInboxに書き留めて自分の作業に戻り、後で「被依頼(自分ボール)」のタスクとして登録する。回答したら内容を作業ログに残して完了する
5. **随時整理**: 手が空いたときにInboxを整理し、案件・期限を割り当てて正式なタスク(被依頼含む)・課題にする
6. **打合せ前の確認**: 相手別ビューで「この相手に何を待っているか/何を待たせているか」をまとめて確認してから打合せ・電話に臨む
7. **過去の経緯の検索**: 似た質問が来たら、キーワード検索で過去の回答・作業ログを引いて再利用する
8. **受入テストの進捗管理**: 環境整備(自部門作業)とテストデータ作成・検証(業務部門作業)をタスクとして一括登録し、定例・メールで聞いた進捗を自分で更新する。部門別の消化状況をフィルタ・集計で確認し、遅れている部門をフォローする
9. **週次報告**: 1週間の記録から自動生成された週報下書きを整え、Excelで出力する

## 3. スコープ

### 3.1 対象とするもの

- 複数プロジェクト(4〜10件並行)の登録・切替・横断ビュー
- Inbox(2秒で書き留め、後から整理)
- タスク管理(WBS的な分解、期限、進捗、担当(自分/他部門))
- 受入テストの準備の進捗管理(環境整備、業務部門によるテストデータ作成・検証。担当付きタスク+一括登録+担当別集計で実現)
- **ボール管理(双方向・タスクに統合)**(返答待ち: 催促候補の自動抽出/被依頼: 要回答の自動抽出。独立した「依頼」は設けず二重管理を防ぐ)
- 相手マスタと相手別ビュー
- 「今日」ビュー(自動候補+手動ピックで今日やることを管理)
- 横断キーワード検索(タスク(作業ログ含む)、課題、Inbox)
- 案件ホーム(案件ごとの資料・フォルダ・URLのリンク集。ローカルパスはツール経由で開く)
- 案件メモ(タスクに紐づかない案件の記録: 打合せの決定事項、経緯、方針など)
- 回答管理(回答済みの問い合わせをQ&A形式で参照・検索)
- 判断保留箱(すぐ答えられない問い合わせの保留と確認事項の整理)
- 報告資料モード(実績・課題・予定・リスクのワンボタン表示とCopilot用テキストコピー)
- スケジュールの可視化(ガントチャート、マイルストーン)
- 課題・リスク管理(起票〜クローズまでの追跡)
- ダッシュボード(横断ビュー)
- 週報の自動下書き生成・編集・Excel出力
- Excel入出力、JSONバックアップ
- タスクトレイ常駐とWindows通知(開き忘れ対策)

### 3.2 対象としないもの(スコープ外)

- **保守運用の定常業務**(月次作業・定期報告など。既存の仕組みで運用中のため)
- 繰り返しタスクの自動生成
- 本格的なナレッジベース・文書管理(回答管理(F14)は被依頼タスクの回答概要を一覧・検索する軽量な仕組みに留める。文書そのものは既存の共有フォルダ等に置き、リンクで参照する)
- 複数人での共有・同時編集・権限管理
- サーバ・クラウドへのデータ保存(データはすべてローカル)
- 工数実績の記録・集計
- メール・Teams等からの自動取り込み(転記は手動。将来検討)
- スマートフォン対応(PC利用のみ)

## 4. 機能要件

### F1. プロジェクト管理

| ID | 要件 | 優先度 |
|---|---|---|
| F1-1 | プロジェクトを複数登録できる(名称、期間、状態、概要、依頼元/ベンダー) | 必須 |
| F1-2 | プロジェクトの状態を管理できる(計画中/進行中/保留/完了) | 必須 |
| F1-3 | プロジェクト単位の表示と、全プロジェクト横断の表示を切り替えられる | 必須 |
| F1-4 | 完了プロジェクトをアーカイブし、通常表示から除外できる | 推奨 |

### F2. Inbox(クイック書き留め)

| ID | 要件 | 優先度 |
|---|---|---|
| F2-1 | どの画面からでも1操作で入力欄にフォーカスし、1行メモ+Enterで書き留められる(目標2秒) | 必須 |
| F2-2 | Inboxの項目を後から「タスク(被依頼・返答待ち含む)」「課題」「案件メモ」に変換し、案件・期限・相手等を割り当てられる | 必須 |
| F2-3 | 未整理のInbox件数をダッシュボード・ヘッダーに常時表示する | 必須 |
| F2-4 | 不要になったメモはワンクリックで破棄できる | 必須 |

### F3. タスク・進捗管理

| ID | 要件 | 優先度 |
|---|---|---|
| F3-1 | タスクを登録できる(件名、詳細、期限、優先度、状態、進捗率) | 必須 |
| F3-2 | タスクの状態を管理できる(未着手/進行中/**返答待ち**/完了/保留)。「返答待ち」はボールが相手にあることを表す(詳細はF4) | 必須 |
| F3-3 | タスクに親子関係を持たせられる(大タスク→サブタスク) | 必須 |
| F3-4 | 一覧をフィルタ・ソートできる(状態、期限、優先度、プロジェクト、担当) | 必須 |
| F3-5 | 期限切れ・期限間近(3日以内)のタスクを視覚的に強調する | 必須 |
| F3-6 | タスクにメモ・作業ログを時系列で残せる(週報の自動下書きの材料になる) | 必須 |
| F3-7 | タスクに担当を設定できる(既定=自分。相手マスタの部門・担当者を指定可)。他部門担当のタスクは「実行は他者・進捗管理は自分」を表す | 必須 |
| F3-8 | **一括登録**: 項目名を複数行貼り付けると、項目ごとに工程テンプレート(例:「作成」「検証」の2子タスク)を適用した親子タスクを一括生成できる。担当・期限もまとめて設定できる(受入テストのテストデータ50〜300項目を想定) | 必須 |
| F3-9 | タスク一覧を担当別にグループ表示し、担当ごとの消化状況(完了n/全m件)を集計表示できる | 必須 |
| F3-10 | タスク一覧・ボール一覧・ダッシュボードから1操作で「今日やる」に追加/解除できる | 必須 |
| F3-11 | タスク・課題に**関連リンク**(ファイルパス、フォルダ、URL、メール参照)を複数添付でき、クリックで開ける(ローカルパスはサーバ経由でエクスプローラー/関連付けアプリを起動) | 必須 |

### F4. ボール管理(本ツールの中核機能。タスクに統合)

独立した「依頼」エンティティは設けず、**タスクに統合する**(二重管理の防止)。
1つの仕事のボールの往復(自分で準備→相手に依頼→回答受領→続行→完了)を1レコードで追跡する。

- **返答待ち(ボール=相手)**: タスクの状態を「返答待ち」にし、相手を指定した状態
- **被依頼(ボール=自分)**: 相手から受けた仕事であることを示す属性を持つタスク

| ID | 要件 | 優先度 |
|---|---|---|
| F4-1 | タスクを「返答待ち」に切り替える際、相手(必須)・待ち開始日(既定=当日)・回答希望日(任意)を記録できる | 必須 |
| F4-2 | 【返答待ち】待ち開始日(または最終催促日)から**5営業日**回答がないものを「催促候補」として自動抽出し、ダッシュボードに表示する | 必須 |
| F4-3 | 【返答待ち】「催促した」を記録でき(催促日を履歴として保持)、催促からの経過営業日で再度警告する | 必須 |
| F4-4 | 回答希望日が設定されている場合は、希望日超過を優先して警告する | 必須 |
| F4-5 | 【被依頼】相手から受けたタスクには依頼元(相手)と発生日を記録でき、発生日から**3営業日**完了しないものを「要回答」として自動抽出し、ダッシュボードに表示する | 必須 |
| F4-6 | 警告までの経過日数は設定画面で変更できる(既定: 返答待ち=5営業日、被依頼=3営業日) | 必須 |
| F4-7 | 相手をマスタ登録できる(名前、所属(ベンダー名/部署名)、メモ)。ボールの相手とタスクの担当(F3-7)で共用する | 必須 |
| F4-8 | **相手別ビュー**: 相手を選ぶと「待っているもの(返答待ち)/待たせているもの(被依頼)」を一覧表示する。打合せ・電話前の確認に使う | 必須 |
| F4-9 | 経過日数は営業日(土日・祝日を除く)で計算する | 必須 |
| F4-10 | 回答受領時は内容をタスクの作業ログ(F3-6)に記録して状態を進行中に戻す(または完了する)。回答内容は後から横断検索で引ける | 必須 |
| F4-11 | **ボール一覧**: 返答待ち・被依頼のタスクを経過営業日つきで一覧表示する(「相手ボール/自分ボール」タブ切替) | 必須 |

|---|---|---|
| F5-1 | タスクの期間(開始日〜期限)をガントチャートで表示できる | 必須 |

| F5-5 | ガント上でバーをドラッグして日程を変更できる | 任意 |

### F6. 課題・リスク管理

| ID | 要件 | 優先度 |
|---|---|---|
| F6-1 | 課題を起票できる(件名、内容、発生日、期限、重要度、対応者、状態) | 必須 |

| F6-3 | 課題に対応履歴を時系列で記録できる | 必須 |

### F7. ダッシュボード

| ID | 要件 | 優先度 |
|---|---|---|
| F7-1 | 起動時に横断ダッシュボードを表示し、最上段に**「今日」セクション**を置く | 必須 |
| F7-2 | 「今日」セクションには、自動候補(期限切れ/今日期限/催促候補/要回答)が必ず表示され、加えて任意のタスクを手動で「今日やる」に追加できる。残件数を表示する | 必須 |
| F7-3 | 手動ピックは当日限り有効とする。未完了のまま日をまたいだ項目は翌朝「昨日の残り」として表示し、1操作で再ピック/解除できる | 必須 |
| F7-4 | 「期限間近(3日以内)」のタスクを一覧表示する | 必須 |
| F7-5 | 未整理のInbox件数と内容を表示する | 必須 |
| F7-6 | 未対応・対応中の課題件数と要注意課題を表示する | 必須 |
| F7-7 | プロジェクトごとの進捗サマリ(タスク消化状況・返答待ち件数)を表示する | 推奨 |
| F7-8 | 直近のマイルストーンを表示する | 推奨 |

### F8. 報告(報告資料モード・週報)

記録(タスク・課題・ボール)から報告内容を自動生成する共通エンジンを持ち、3つの出力形態を提供する:
①画面即時表示(報告資料モード) ②Copilot用テキストコピー ③Excel出力(週報)。

| ID | 要件 | 優先度 |
|---|---|---|

## 8. 技術方針(案)

| 項目 | 方針 | 理由 |
|---|---|---|
| 構成 | Go製の単一exe。UI資材(HTML/CSS/JS)は `go:embed` でexeに埋め込み、標準ライブラリ `net/http` で配信 | ビルド成果物が1ファイルで配布・起動が容易。ランタイム依存なし |
| サーバ | localhost のみで待ち受ける軽量HTTPサーバ+データ読み書き用のREST API | ブラウザからファイル保存を可能にする最小構成 |
| 常駐・通知 | タスクトレイ常駐(systray系ライブラリ)+Windowsトースト通知。スタートアップ登録はスタートアップフォルダへのショートカット作成で実現 | 「開き忘れ」対策をツール外(OS通知)から行うため |
| フロントエンド | 素のJavaScript(フレームワーク不使用)、必要なら軽量ライブラリを同梱 | 外部依存を排除しオフライン動作を保証するため |
| Excel入出力 | Go側で excelize ライブラリを使用(エクスポート=ダウンロード、インポート=アップロード) | .xlsx の読み書きをサーバ側で完結でき、ブラウザ側の実装が簡素になる |
| 営業日計算 | 土日+同梱の祝日データ(年次更新)+ユーザー定義休日で判定 | 催促候補の精度確保。オフラインでも動作 |
| ガントチャート | 自前実装(SVGまたはHTML/CSS) | 外部依存を避けつつ、必要十分な表示に絞るため |
| ローカルパスを開く | リンクのクリックでサーバ側APIを呼び、OSの関連付け(explorer/start)でファイル・フォルダを開く | ブラウザは `file://` を直接開けないため。localhost限定なので安全性も確保できる |
| Copilot連携 | API連携はせず、報告内容を構造化テキストとしてクリップボードにコピーする方式 | 完全オフライン要件(N3・N6)を守りつつ、M365 Copilot等での清書を可能にする |
| データ保存 | `data/pjkr.json` への自動保存(一時ファイル経由の安全な書き込み+世代バックアップ) | ブラウザ設定に依存せず、ファイルコピーだけでバックアップ・PC移行が可能 |
| 起動 | exe起動時にサーバを立ち上げ、既定ブラウザで `http://localhost:8760` を自動オープン。以後はトレイに常駐 | 利用開始までの手数を最小化 |

### ディレクトリ構成(配布時)

```
PJKR/
 ├─ pjkr.exe        ← ダブルクリックで起動、トレイ常駐
 └─ data/
     ├─ pjkr.json   ← データ本体(自動保存)
     └─ backup/     ← 世代バックアップ(自動)
```

## 9. 開発フェーズ(案)

痛点の大きい順(返答待ち追跡 → 把握の速さ → 報告負担)に価値を届ける構成とする。

| フェーズ | ねらい | 含まれる機能 |
|---|---|---|
| Phase 1 (MVP) | 書き留めと日常のタスク運用を開始する | サーバ基盤(起動・自動保存・世代バックアップ)、プロジェクト管理、Inbox、タスク管理(担当・関連リンクを含む)、案件ホーム(リンク集+案件メモ+ローカルパスを開く仕組み)、ダッシュボード(「今日」ビュー: 自動候補(期限系)+手動ピック) |
| Phase 2 | **中核のボール管理**と気づきの仕組み | ボール管理(タスク統合: 返答待ち・被依頼、催促候補・要回答、相手マスタ、催促履歴、相手別ビュー、営業日計算)、判断保留箱、回答管理、「今日」ビューへの催促候補・要回答の合流、横断検索、課題・リスク管理、タスク一括登録・担当別集計(受入テスト対応)、トレイ常駐・朝の通知・スタートアップ登録 |
| Phase 3 | 報告業務の効率化 | 報告資料モード(即時表示)、Copilot用テキストコピー、週報(自動下書き→編集→Excel出力・過去参照)、タスク/課題/ボール/Q&AのExcelエクスポート、Excelインポート、JSONリストア |
| Phase 4 | 可視化と改善 | ガントチャート、マイルストーン、任意要件(ドラッグ日程変更、列カスタマイズ、日中通知、共通リンク等) |

> ガントチャートをPhase 4に置いた理由: ヒアリングの結果、日々の運用の中心は「期限・ボール・課題の追跡」であり、
> 工程の可視化は計画立案時に使う補助機能と位置づけたため。優先度を上げたい場合は入れ替え可能。
